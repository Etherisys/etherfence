# Phase 0 Research: Posture Score Risk Separation

All unknowns below were resolved by reading the current implementation directly (no external unknowns — this is a self-contained Rust workspace change). Each entry follows Decision / Rationale / Alternatives considered.

## 1. How to make `EF-MCP-000`/`EF-MCP-004` stop scoring

**Decision**: Reclassify `Severity` of `EF-MCP-000` (`McpServerConfigured`) and `EF-MCP-004` (`ExposedMcpEnvironment`) from `Low` to `Info` in their `FindingTemplate` in `crates/etherfence-detectors/src/lib.rs`. Additionally add a new `FindingCategory` enum (`Inventory`, `Informational`, `Risk`) as an explicit field on `Finding`, and gate `PostureSummary::from_findings`'s score/high/medium/low/info computation on `category == Risk` rather than relying implicitly on `severity != Info`.

**Rationale**: `Severity::Info` is already excluded from the score formula and from `priority_risks`/`recommended_actions` selection today (`crates/etherfence-core/src/lib.rs:626-665`), so the severity change alone would already stop these two finding kinds from scoring. However, the spec (FR-007) requires category to be the *explicit, independent* scoring gate — severity should represent only risk magnitude, not "does this count." Making the gate explicit prevents future bugs where someone adds a new Info-severity finding that should still score, or a new non-Info-severity finding that shouldn't (e.g., a future zero-weight inventory fact at `Low`). It also gives the human-output layer (four-way grouping) a single authoritative field to group by, instead of re-deriving groups from ad hoc ID checks (as today's `if finding.id == "EF-MCP-000"` special-case in `verbose.rs:416` already shows is fragile).

**Alternatives considered**:
- *Severity-only change, no category field*: Simpler, smaller diff, and would achieve identical scoring behavior today (since Info already excludes). Rejected because it doesn't satisfy FR-007's explicit requirement, doesn't fix the fragile ID-based special-casing in `verbose.rs`, and gives the four-way human-output split no structural backing (would require inventing severity-plus-ID heuristics per renderer).
- *New `Severity::None` variant instead of a separate category*: Rejected — conflates a magnitude scale with a scoring-eligibility flag, contradicts FR-007's explicit requirement that severity stay pure risk magnitude, and would require renumbering/reordering the `Severity` enum (used for SARIF `level` mapping and sort order), a much larger blast radius than adding an independent field.

## 2. Evidence field-labeling scheme

**Decision**: Every evidence string keeps its existing `Vec<String>` shape, but is normalized to a consistent `field=value` format across all detectors: `server=<name>`, `command=<value>`, `args[<i>]=<value>` (index included so distinct arguments never collapse into ambiguous identical evidence), `url=<value>`, and `env=<name>` for environment-variable-name matches. `EF-MCP-000`'s evidence already uses `server=`/`command=`/`url=` (`crates/etherfence-detectors/src/lib.rs:250-259`) — this becomes the baseline convention extended to the other detectors (`EF-MCP-001/002/003` today emit bare unlabeled matched values via the `values()` helper, and `EF-MCP-004`/`EF-SEC-001` emit bare environment variable names).

**Rationale**: Satisfies FR-005 (evidence must name the specific field) with the smallest possible schema footprint — no new field, no new type, reuses a convention the codebase already partially established. Never includes a redacted secret value; only the variable *name*/matched pattern is ever placed in evidence, exactly as today.

**Alternatives considered**:
- *Structured `{key, value}` evidence objects* (mirroring `etherfence-setup`'s trust-indicator evidence shape documented in `docs/json-schema.md:281`): more rigorously typed, but changes `Finding.evidence`'s type from `Vec<String>` to `Vec<EvidenceEntry>`, a much larger blast radius across fingerprinting, JSON/Markdown/SARIF renderers, `docs/examples/ci/baseline.json`, and every existing test that indexes into `finding.evidence` as strings. Rejected under Scope Discipline (Principle X) — the string-label convention meets FR-005/FR-006 without a type change.

**Known compatibility consequence (accepted, documented)**: `finding_fingerprint()` (`crates/etherfence-core/src/lib.rs:792-810`) hashes sorted/deduped `evidence` — changing evidence text for `EF-MCP-001/002/003/EF-MCP-004/EF-SEC-001` changes those findings' fingerprints. This is why both `ef-scan-report` and `ef-baseline` schema versions bump (see #4) and why `read_baseline()` already fails closed on schema-version mismatch (`crates/etherfence-cli/src/main.rs:2881-2889`) — an old baseline file is rejected outright with an explicit "regenerate it with `--write-baseline`" message rather than silently mismatching fingerprints. This is called out prominently in the CHANGELOG.

## 3. Where `ProtectionCoverage` fits the four-way human-output split

**Decision**: No change needed to `ProtectionCoverage` itself. Confirmed by reading `crates/etherfence-core/src/lib.rs:749-758` and `crates/etherfence-cli/src/main.rs:2339-2345,2569-2571`: it is already a fully separate top-level `ScanReport.protection_coverage: Option<ProtectionCoverage>` field, populated only under `--policy`/`--policy-profile`, entirely independent of the `Finding` list. It already renders as its own "Protection coverage" section in both concise and verbose human output.

**Rationale**: The fourth category required by the spec ("protection or policy coverage") already exists as a distinct, correctly-separated concept; this feature only needs to keep it visually distinct alongside the three new/adjusted Finding-derived groups, not redesign it.

## 4. Schema version bumps

**Decision**: Bump `ef-scan-report/v0.1.2` → `v0.1.3` (new `Finding.category` field + severity value change for two finding kinds + evidence format normalization for five finding kinds). Bump `ef-baseline/v0.1.3` → `v0.1.4` (baseline files embed full `Finding` structs — `BaselineFile.findings: Vec<Finding>` at `crates/etherfence-core/src/lib.rs:783-790` — so the same `Finding` shape change propagates into the baseline schema).

**Rationale**: Principle VI requires an explicit version bump for any incompatible field-semantics change. `category` is additive (new field), but the severity/evidence changes for existing finding IDs are semantic changes to existing fields — bump required either way per the constitution's explicit rule ("Any incompatible change to field names, field semantics, required fields... MUST bump the schema version"). `read_baseline()` already fails closed on a schema-version mismatch, so old baseline files are rejected with a clear regenerate instruction instead of silently misbehaving — this is the existing, correct fail-closed mechanism, not new code.

**Alternatives considered**: Leaving `ef-baseline` at `v0.1.3` unchanged and letting old/new `Finding` shapes coexist informally — rejected because `BaselineFile` directly embeds `Finding`, so the shape changed regardless of whether the version string does; not bumping would silently violate Principle VI.

## 5. `Finding.category` backward-compatible deserialization

**Decision**: `category: FindingCategory` gets `#[serde(default)]`, with `FindingCategory::Risk` as the `#[default]` variant (mirroring the existing `PolicyStatus::NotApplicable` `#[default]` pattern at `crates/etherfence-core/src/lib.rs:424-431`).

**Rationale**: Defaulting unknown/legacy data to `Risk` (the most conservative/scored option) is consistent with Principle I (deny-by-default / most-conservative default on ambiguity) — if a `Finding` value ever arrives without a category (e.g., a hand-constructed test fixture that predates this change, or a hypothetical future consumer), it is treated as actionable rather than silently excluded from scoring. In practice this only matters for `#[serde(default)]`'s JSON-deserialization fallback; every real construction site in `etherfence-detectors`/`etherfence-policy` sets `category` explicitly, and (per decision #4) old baseline files are rejected before deserialization would even reach this fallback in the one place it could matter (`read_baseline`'s schema-version gate runs after parsing, so the default is truly only a defensive belt-and-suspenders, not the primary compatibility mechanism).

## 6. `PostureSummary` count-field semantics after the change

**Decision**: `PostureSummary.active_findings`/`high`/`medium`/`low`/`info` (and the `score`/`priority_risks`/`recommended_actions` derived from them) are computed over `category == Risk` active findings only — a narrower population than today (today: all active findings regardless of category/severity, though `Info` severity was already separately excluded from the priority/action lists). The top-level `report.summary.*` (`Summary::from_counts`, `crates/etherfence-core/src/lib.rs:483-503`) is unchanged code and continues to count *all* displayed findings by severity regardless of category — it naturally reflects the new severities once detector templates change, with no code change required.

**Rationale**: `posture` is documented as "deterministic advisory score... derived from displayed active findings" — after this feature, that phrase should mean scored-risk findings specifically, matching what the score itself measures. `report.summary` remains the raw, complete severity breakdown (now naturally showing fewer `Low`/more `Info` after the two finding kinds' severity changes) — this is the correct place for an honest, unfiltered "what did we see" count, while `posture.*` is the correct place for "what should you act on."

**Alternatives considered**: Keeping `posture.*` counts over all active findings (unchanged population) while only gating the `score` integer itself on category — rejected as internally inconsistent (a `posture.info` count that includes non-scoring findings while `posture.score` excludes them would misleadingly suggest those findings partially matter to posture).

## 7. Human-output four-way grouping mechanics

**Decision**:
- **Concise (`render_scan_summary`, `crates/etherfence-cli/src/main.rs`)**: add two new small, clearly-headed sections ("Inventory observations" and "Informational findings") between the existing "Clients" and "Protection coverage" sections, each a short summary line (not a full per-finding dump, consistent with the concise view's existing "executive summary" design intent per its own doc comment at `main.rs:2420-2422`). The existing "Priority findings" section already becomes risk-only automatically once `PostureSummary` is category-gated (decision #6) — no rename needed, it already unambiguously means "scored risk." "Protection coverage" is unchanged (decision #3).
- **Verbose (`render_scan_verbose`/`render_findings`, `crates/etherfence-cli/src/verbose.rs`)**: change the per-finding badge in `render_findings` (currently keyed only on `finding.severity`) to key on `finding.category` first — `Inventory` → a new "OBS " badge, `Informational` → keep "INFO" badge, `Risk` → keep existing HIGH/MEDIUM/LOW/INFO-by-severity badges. Simplify `render_consolidated_recommendations`'s existing `if finding.id == "EF-MCP-000" { continue }` special-case (`verbose.rs:416`) to a general `if finding.category != FindingCategory::Risk { continue }` check — this also naturally removes `EF-MCP-004` and `EF-TIRITH-*` from ever appearing as numbered "recommended actions," which is correct per FR-001/FR-002 (they are not actionable remediations) but is a behavior change to an existing test (`scan_verbose_consolidated_excludes_context_and_orders_by_severity` currently asserts `EF-MCP-004` appears there — this assertion must be updated, not preserved, since keeping it would contradict the feature's own required outcome).
- **Markdown (`crates/etherfence-report/src/lib.rs`, both `append_human_findings`/`to_human` and the Markdown `## Findings` section in `to_markdown`)**: change the grouping loop from `for severity in Severity::ORDERED_DESC` to a two-level grouping — `for category in [Inventory, Informational, Risk]` outer, `for severity in Severity::ORDERED_DESC` inner within `Risk` — with a category heading, matching the same four-way split for consistency across all human-readable formats (even though `to_human`/`append_human_findings` is not currently wired into the CLI — confirmed via `grep`, its only callers are the `etherfence-report` crate's own unit tests — it is still public API and should not silently diverge from the CLI's actual verbose renderer).

**Rationale**: Reuses the existing section/heading pattern already established by "Protection coverage," keeps the concise view genuinely concise (a full per-finding inventory dump in the default view would regress the "executive summary" design principle from `specs/007-scan-posture-stabilization`), and fixes the pre-existing fragile ID-string special-case in `verbose.rs` with a general category check.

## 8. CHANGELOG placement

**Decision**: Rename the existing `## [Unreleased]` section (currently holding the merged, not-yet-version-bumped "post-v1.7.3 security & correctness review" content) to `## [1.7.4] - 2026-07-13`, and add a new subsection under it (e.g. "### Changed" / "### Scoring") describing the risk-separation change, plus a fresh empty `## [Unreleased]` placeholder above it for future work.

**Rationale**: `Cargo.toml` is still at `1.7.3` even though the post-v1.7.3 review already merged (it was a security/correctness patch that intentionally did not warrant a version bump, per its own CHANGELOG note: "No schema version, scoring, or scan-discovery change"). This feature is the first change since that review to warrant a version bump, so both bodies of work ship together under `1.7.4`. This is "Keep a Changelog"-style promotion of the `Unreleased` placeholder, not a rename of an already-cut version heading, so it does not violate the project's "never rename an existing version heading" rule (per CLAUDE.md).
